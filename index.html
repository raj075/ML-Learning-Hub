🧹 Data Preprocessing

Getting raw data ready for machine learning models.

💡 Preprocessing is not glamorous — but it's the difference between a model that works and one that doesn't.

Why Preprocessing?

Most ML algorithms only understand numbers, can't handle missing values, and are sensitive to scale differences. Preprocessing fixes all of this before training begins.

Discrete Data

Data from a fixed set of categories (text labels). Needs to be converted to numbers before a model can use it.

Continuous Data

Real-valued numbers (prices, heights). Needs scaling so different features are on the same range.

One-Hot Encoding

Converts a categorical column into multiple binary (0/1) columns — one per unique category. Only one column is "on" at a time.

Originalis_hotis_coldis_lowis_midis_high
hot, mid10010
cold, high01001

encoder = OneHotEncoder()
encoder.fit(data)          # learn categories
encoder.transform(data)   # convert to binary vectors

Missing Data — Imputation

  • Drop the row — only if missing is random and you have enough data
  • Fill with mean — good default for symmetric data
  • Fill with median — better when outliers are present
  • Interpolate — for time series, fill using neighbouring values
data.dropna(subset=['review_scores_rating'], inplace=True)
# Only drop rows where this specific column is NaN

Outlier Detection — Z-Score

A z-score measures how many standard deviations a value is from the mean. Values with |z| > 3 are flagged as outliers.

def zscore(sample):
    return (sample - sample.mean()) / sample.std()

outliers = data[zscore(data['price']).abs() > 3]

⚙️ Feature Engineering

Creating better inputs so your model can learn more.

The Core Problem

Linear models can only fit straight lines. If the relationship between your input and output is curved, a linear model will always underperform — no matter how much data you have.

Feature engineering is the solution: create new features from your existing ones so the model can capture non-linear patterns while still being a linear model.

Key Insight

A linear model is linear in its weights, not in its features. You can feed it x², log(x), or sin(x) — it will still use them linearly, but the result is a curve.

📉 Linear Regression (Gradient Descent)

Finding the line of best fit through iteration.

The Goal

Predict a continuous output from one or more inputs by finding a straight line (or hyperplane) through the data. The model: ŷ = w·x + b

Weight (w)

The slope. How much does ŷ change per unit increase in x?

Bias (b)

The intercept. The baseline prediction when x = 0.

Mean Squared Error (MSE)

The cost function — measures how wrong the model is. Lower = better.

MSE = (1/N) × Σ(yᵢ − ŷᵢ)²

Square each error (prevents cancellation), average across all N samples.

Gradient Descent Algorithm

Instead of trying random values, we use calculus to find which direction reduces MSE fastest and step in that direction repeatedly.

ElementDescriptionTypical value
GradientDirection to step (from derivative of MSE)Calculated each iteration
Learning rate (η)How far to step each iteration0.0001
Starting pointInitial guess for w and bw=0, b=0
Stopping criterionWhen to stop100,000 iterations

Gradient Descent in Python

w, b = 0, 0
eta = 0.0001

for i in range(100000):
    y_pred = w*x + b
    D_w = (-2/n) * sum(x * (y - y_pred))
    D_b = (-2/n) * sum(y - y_pred)
    w = w - eta * D_w
    b = b - eta * D_b

R² — Coefficient of Determination

R² = 1 − SSE / SST
R² valueMeaning
1.0Perfect fit
0.9Model explains 90% of variation — very good
0.0No better than predicting the mean every time
< 0Worse than the mean — something is wrong

🔢 Linear Regression — OLS Method

Finding the exact optimal parameters in one calculation.

OLS vs Gradient Descent

Gradient DescentOLS
ApproachIterative — thousands of stepsDirect — one formula
ResultApproximate (converges)Exact optimal
SpeedSlow for large dataFast for moderate data
RequiresLearning rate tuningMatrix to be invertible

The OLS Formula

β* = (XᵀX)⁻¹ Xᵀy

One equation. Gives you the exact optimal weights in a single calculation.

Xᵀ

Transpose — flip rows and columns of the data matrix X.

(XᵀX)⁻¹

Matrix inverse — the matrix equivalent of dividing.

OLS in Python

import numpy as np

X = np.hstack((np.ones((n, 1)), x))  # add bias column of 1s

beta = np.linalg.inv(X.T @ X) @ np.dot(X.T, y)

Y_pred = X @ beta  # predictions

Using Statsmodels for Hypothesis Testing

import statsmodels.api as sm

X = sm.add_constant(x)     # adds column of 1s
results = sm.OLS(y, X).fit()
print(results.summary())   # p-values, R², F-stat

Reading the Summary Table

  • coef — the learned β weights
  • P>|t| — p-value: if < 0.05, the feature is statistically significant
  • R-squared — proportion of variance explained
  • F-statistic — is the overall model better than just predicting the mean?

Regularisation

Add a penalty to the cost function to discourage large weights and prevent overfitting.

Ridge (L2)

Cost = MSE + λ × Σβⱼ²
  • Shrinks all weights towards zero
  • Never removes a feature entirely
  • Use when all features matter

Lasso (L1)

Cost = MSE + λ × Σ|βⱼ|
  • Can push weights to exactly zero
  • Automatic feature selection
  • Use when many features are irrelevant
from sklearn.linear_model import Ridge, Lasso

ridge = Ridge(alpha=0.5)   # lambda = 0.5
lasso = Lasso(alpha=0.5)   # lambda = 0.5

ridge.fit(X_train, y_train)
ridge.predict(X_test)

📋 Assumptions of Linear Regression

When can you trust your model? Know the rules it relies on.

The Error Term

Every linear model has: y = wx + b + ε. The error ε represents everything the model cannot explain. If the model is correct, ε is pure random noise — no pattern at all. The 5 assumptions describe what ε should look like.

The 5 Assumptions

1. Linearity

Each feature has a constant, linear effect on the output regardless of its value.

Check: scatter plot each feature vs target. Fix: feature engineering.

2. Homoscedasticity (Constant Variance)

The spread of errors is the same everywhere — not larger for bigger predictions.

Check: residual vs prediction plot — should be a horizontal band. Fix: log-transform target.

3. Independence

Each observation's error is independent of every other observation's error.

Violated by: time series data, spatial data. Fix: use time-series models.

4. Normality

The error terms follow a normal distribution.

Check: Shapiro-Wilk test (p < 0.05 = non-normal). Note: less critical for pure prediction tasks.

5. No Multicollinearity

Input features should not be highly correlated with each other.

Check: correlation heatmap — values above 0.7 or below −0.7 are problematic. Fix: remove one feature, use PCA, or use Lasso.

Diagnosing with Residual Plots

What you see in residual plotAssumption violated
Random horizontal bandAll good ✓
Clear curve or waveLinearity
Fan/funnel shape (spread grows)Homoscedasticity
Repeating patternIndependence
Heavy tails in distributionNormality

🔧 Scikit-Learn for Machine Learning

The professional Python ML library — one API for everything.

Why sklearn?

sklearn contains hundreds of ML algorithms all sharing the same interface. This means you can swap models with almost no code changes — the same 5-step pattern works for everything.

The Universal Pattern

# 1. Import
from sklearn.linear_model import LinearRegression

# 2. Create (set hyperparameters here)
model = LinearRegression()

# 3. Fit on training data
model.fit(X_train, y_train)

# 4. Predict on new data
y_pred = model.predict(X_test)

# 5. Evaluate
score = model.score(X_test, y_test)

Naming Conventions

Classes — PascalCase

LinearRegression, StandardScaler, Ridge
Objects with memory — create an instance, call methods.

Functions — snake_case

train_test_split, make_regression, mean_squared_error
One-shot actions — call and get a result.

🔭 Dimensionality Reduction — PCA

Finding hidden structure in unlabelled data.

Unsupervised Learning

No target variable. No predictions. Instead, the question is: "What is the underlying structure of my data?"

Supervised (Lessons 5-8)

Inputs → predict Target. Labelled data required.

Unsupervised (PCA)

Find structure in all variables together. No labels needed.

The Core Intuition

If French and Spanish scores are correlated, there's likely a hidden latent factor — language aptitude — driving both. PCA finds these hidden directions automatically.

The key rule: the best direction to summarise data is the one that maximises the variance of the projections. High variance = more information kept.

Principal Components

TermMeaning
Principal directionA direction through the data ordered by variance explained (most first)
Component scoresWhere each data point lands when projected onto a principal direction
Explained variance ratio% of total variance captured by k components

PCA in Sklearn

from sklearn.decomposition import PCA

pca = PCA(n_components=2)
pca.fit(X_train)                     # fit on training data only!

X_train_reduced = pca.transform(X_train)
X_test_reduced  = pca.transform(X_test)

print(pca.explained_variance_ratio_)  # e.g. [0.72, 0.15]
print(pca.components_)               # the principal directions

How Much to Reduce?

Plot explained variance per component (scree plot) and look for the "elbow" — the point where adding more components gives diminishing returns.

  • Explained variance ≈ 0.5 → little to gain, data nearly uncorrelated
  • Explained variance ≈ 0.9+ → you've kept most information in far fewer dimensions

When to Use PCA

ReasonExplanation
MulticollinearityPCA components are always uncorrelated by design
Too many featuresCompress d features to k << d without losing much information
VisualisationReduce to 2-3 components to plot and spot clusters
⚠️ Always standardise your data before PCA. Features with larger scales will dominate the variance calculation purely because their numbers are bigger.

L8: PCA — Dimensionality Reduction

Core idea

Find the directions through the data that capture the most variance (information). Project all data onto K of these directions, reducing from d features to K features without losing much information.

TermMeaning
Principal componentA direction ordered by how much variance it captures (PC1 = most, PC2 = second most, …)
Component scoresWhere each data point lands when projected onto a principal direction
Explained variance ratioFraction of total variance captured by each component — check cumulative sum to decide how many to keep
Scree plotPlot of explained variance per component — look for the elbow (point of diminishing returns)

When to use PCA

ReasonWhy PCA helps
MulticollinearityPCA components are always orthogonal (uncorrelated) by design
Too many featuresCompress d features to k << d with minimal information loss
VisualisationReduce to 2–3 components to plot high-dimensional data
SpeedFewer features = faster model training
⚠️ Always StandardScale before PCA. Features with larger scales will dominate the variance calculation — the first component will just track the biggest-scale feature.
from sklearn.decomposition import PCA
from sklearn.preprocessing import StandardScaler

scaler = StandardScaler()
X_scaled = scaler.fit_transform(X_train)     # scale first, always

pca = PCA(n_components=2)
pca.fit(X_scaled)
print(pca.explained_variance_ratio_)          # e.g. [0.72, 0.15]
print(pca.explained_variance_ratio_.cumsum()) # cumulative: [0.72, 0.87]

X_reduced = pca.transform(X_scaled)          # now only 2 features

Foundation Golden Rules

Rules that apply across the whole Foundation chapter
  1. Split before you do anything — preprocessing, scaling, PCA all go inside the CV loop or Pipeline
  2. Fit transformers on training data only — StandardScaler, PCA, OneHotEncoder: fit on train, transform on both
  3. Check class/target distribution before modelling — skewed targets often need log-transform
  4. Check residual plots after fitting — they reveal which of the 5 assumptions are violated
  5. R² alone is not enough — always also look at residual plots; a bad fit can still show a reasonable R²
  6. High training score ≠ good model — always compare train vs test; a large gap means overfitting
  7. Correlation heatmap before OLS — multicollinearity (|r| > 0.7 between features) makes coefficients unreliable
  8. Use sklearn Pipelines — prevents accidental data leakage when combining preprocessing and modelling

🎯 Introduction to Classification

Predicting categories, not numbers — the other half of supervised learning.

Classification vs Regression

Both are supervised learning — both use labeled training data. The difference is the output type.

TaskOutputExampleTest to tell apart
RegressionContinuous numberPredict house price: £342,000Is one prediction "between" two others?
ClassificationDiscrete class labelPredict email: Spam / Not Spam"Spam" is not between "Not Spam" and "Fraud"

Binary vs Multiclass

Binary

Exactly 2 classes — one positive, one negative. E.g. Fraud / Not Fraud, Disease / Healthy.

Multiclass

3 or more classes. E.g. Handwritten digit: 0–9. Needs special strategies (One-vs-Rest, Softmax).

The Decision Boundary

Every classifier, no matter how complex, draws a decision boundary in feature space — a line or surface that separates classes. Points on one side are predicted as class A, the other as class B.

  • Linear boundary — Logistic Regression, linear SVM: a straight line or hyperplane
  • Non-linear boundary — Decision Trees, KNN, SVM (RBF): curves and complex shapes

Even the simplest classifier — "always predict Class A" — has a boundary. It's just infinitely far away.

Two Families of Classifiers

The most important conceptual split in classification. Both families produce decision boundaries, but they get there differently.

Discriminative Models

Learn the boundary directly. Model P(y|x) — the probability of a class given the features.

"Where is the line between the classes?"

  • Cannot generate new data
  • Makes fewer assumptions
  • Better with lots of data
  • More sensitive to outliers

KNNLogistic RegressionDecision Trees

Generative Models

Model what each class looks like first. Learn P(x|y), then derive P(y|x) via Bayes' Rule.

P(y|x) = P(x|y) × P(y) / P(x)

Posterior ∝ Likelihood × Prior. Since P(x) is the same for all classes, we only need the numerator to find the most probable class.

Class Imbalance — Why Enough Data Per Class Matters

Real classification problems are rarely balanced. A class with very few examples means the model learns the quirks of those specific examples, not the general pattern.

The Problem

Fraud is 0.1% of transactions. A model that always predicts "not fraud" gets 99.9% accuracy — and is completely useless. It has learned nothing about fraud.

The Fix

Use stratify=y when splitting, use precision/recall/F1 metrics, consider oversampling the minority class or lowering the decision threshold.

⚠️ Never rely on accuracy alone for imbalanced data. Use precision, recall, and F1 instead — covered in L11.

🎲 Naive Bayes

A generative classifier — it models what each class looks like, then uses Bayes' Rule to decide.

Bayes' Rule — The Foundation

Every generative classifier is built on Bayes' Rule. It says: to find the probability of a class given your data, combine what the data looks like for that class (likelihood) with how common that class is (prior).

P(y | x) = P(x | y) × P(y) / P(x)

Posterior = Likelihood × Prior / Normalisation

TermWhat it meansExample (spam)
P(y | x) — PosteriorProbability of class y given the features x. This is what we want.P(spam | "free money now")
P(x | y) — LikelihoodHow probable is it to see this data if we're in class y?How often does "free money" appear in spam?
P(y) — PriorHow common is class y before seeing any data?30% of all emails are spam
P(x) — NormalisationSame for all classes — we ignore it when comparing.How common is "free money" overall?
Why we ignore P(x)

When choosing the most probable class, we compare P(spam | x) vs P(not spam | x). Both have the same P(x) denominator, so dividing by it doesn't change which one is bigger. We only need to compare the numerators.

The Naive Assumption — Conditional Independence

With multiple features, computing P(x | y) gets complex — you'd need the joint probability of every possible feature combination. Naive Bayes makes a simplifying assumption to make this tractable.

The Assumption

Given the class, each feature is independent of every other feature. So the joint likelihood becomes a simple product of individual likelihoods:

P(x₁, x₂, ..., xₙ | y) = P(x₁|y) × P(x₂|y) × ... × P(xₙ|y)

This is "naive" because it's almost never true in reality — word frequencies in emails are correlated, flight delay types are correlated. But it works surprisingly well anyway, because even imperfect probability estimates tend to rank the right class highest.

Gaussian Naive Bayes — For Continuous Features

When features are continuous numbers (like distance, delay minutes, salary), Naive Bayes models each feature per class as a Gaussian (bell curve). During training it simply calculates the mean and standard deviation of each feature separately for each class.

Training (Fitting)

For each class y and each feature xᵢ, compute:

μᵢʸ = mean of xᵢ for all examples in class y
σᵢʸ = std dev of xᵢ for all examples in class y

Prediction

For a new point, plug each feature value into the Gaussian PDF to get P(xᵢ | y). Multiply across all features × prior. Pick the class with the highest score.

Log Probabilities — Avoiding Numerical Underflow

Multiplying many small probabilities together creates numbers so tiny that a computer rounds them to zero. The fix: use logarithms.

Why logs work

Instead of multiplying: 0.001 × 0.002 × 0.003 × ... → rounds to 0.0
Take log: log(0.001) + log(0.002) + log(0.003) + ... → stays as large negative numbers, safe to compare.

Since log is monotonically increasing, log(A) > log(B) iff A > B — the ranking of classes is preserved. The class with the highest log-probability is the same class with the highest raw probability.

Laplace Smoothing — The Zero Probability Problem

If a word never appeared in training spam emails, P("word" | spam) = 0. That single zero multiplied into the product makes the entire class probability zero — even if all other features strongly point to spam.

The Fix — Add a small count (α) to everything

P(word | class) = (count(word, class) + α) / (total words in class + α × vocabulary size)

Typically α = 1 ("add-one smoothing"). This ensures no probability is ever exactly zero.

⚠️ Laplace smoothing applies mainly to categorical Naive Bayes (discrete word counts). GaussianNB uses continuous distributions which are never exactly zero.

What's happening: We define "major delay" relative to the data itself — flights in the top 30% of positive delays. np.percentile(positive_delays, 70) finds the value below which 70% of actual delays fall. Then we remove ArrDelay from the feature matrix — leaving it in would be data leakage, since it directly defines the label.

What's happening: stratify=major_delay ensures both the training set and test set contain the same proportion of major vs non-major delays. Without it, a random split might put most of the rare "major delay" examples into training, leaving the test set unrepresentative. This is especially important when one class is less frequent.

gnb = GaussianNB()
gnb.fit(X_train, y_train)

What's happening: fit() computes the mean (μ) and standard deviation (σ) of each feature separately for class 0 and class 1. That's it — no iterative optimisation. For 15 features and 2 classes, it calculates and stores 2 × 15 = 30 (μ, σ) pairs. This is why Naive Bayes trains almost instantly.

Strengths and Limitations

Strengths

  • Extremely fast to train — just compute means and variances
  • Works well with small datasets (Gaussian assumptions fill gaps)
  • Handles many features without overfitting
  • Naturally produces calibrated probabilities
  • No hyperparameters to tune

Limitations

  • Independence assumption is almost always violated
  • If features are strongly correlated, predictions become overconfident
  • Gaussian assumption wrong for highly skewed or multimodal features
  • Cannot capture interactions between features
  • Decision boundary is always linear (with Gaussian NB)

📈 Logistic Regression

The simplest classifier — linear boundary, probability output.

Why Not Linear Regression?

You already know classification outputs a class label, not a continuous number. If you fit a straight line to binary labels (0 or 1) you get predictions like 1.7 or −0.3 — meaningless as probabilities. What you actually want is: "given this input, what's the probability it belongs to class 1?" That probability must live in [0, 1]. Linear regression can't guarantee that. Logistic regression can.

The Sigmoid Function

Logistic regression squashes any real number into (0, 1) using the sigmoid. The input z is the same linear combination from linear regression:

σ(z) = 1 / (1 + e−z)   where z = wᵀx + b

Always outputs between 0 and 1. Interpreted as P(y = 1 | x).

z valueσ(z) outputWhat it means
z = 00.5Exactly on the decision boundary — maximum uncertainty
z → +∞→ 1Very strong signal for class 1
z → −∞→ 0Very strong signal for class 0

Thresholding — Probability → Class

P(y=1) ≥ 0.5 → class 1

Default threshold. Since σ(z) ≥ 0.5 iff z ≥ 0, the decision boundary is simply wᵀx + b = 0.

Tuning the threshold

Lower it → catch more positives (higher recall, more false alarms). Raise it → be more certain before predicting positive (higher precision). Covered in L11.

Loss Function — Log Loss (Binary Cross-Entropy)

You can't use MSE here — the loss surface becomes non-convex and gradient descent gets stuck in local minima. Instead logistic regression uses binary cross-entropy:

L = −[y·log(ŷ) + (1−y)·log(1−ŷ)]

Heavily penalises confident wrong predictions. Predicting P=0.99 when truth is 0 → loss → ∞.

y=1 and ŷ≈1

Loss ≈ 0 — correct and confident. Good.

y=1 and ŷ≈0

Loss → ∞ — confident and completely wrong. Heavily penalised.

Training — Gradient Descent

The update rule looks identical to linear regression (L3) — the difference is hidden inside how ŷ is computed (sigmoid vs raw dot product):

# Gradient of log-loss with respect to weights:
# ∂L/∂w = (1/m) · Xᵀ(ŷ − y)
#
# Update rule:
# w = w − α · ∂L/∂w

Regularization — Preventing Overfitting

Without regularization, weights can grow to ±∞ — the sigmoid becomes a step function, perfectly confident on training data but terrible on unseen data. Regularization adds a penalty for large weights.

L2 — Ridge

Loss + λ·Σwᵢ²
Squaring large weights penalises them hard. All weights shrink toward zero, none reach exactly zero. Model still uses all features, just modestly.

L1 — Lasso

Loss + λ·Σ|wᵢ|
Linear penalty has a geometric property that drives some weights to exactly zero — automatic feature selection. Use when many features are likely irrelevant.

The Three sklearn Parameters

ParameterWhat it controlsPractical notes
C=1.0Regularization strength: C = 1/λ. Lower C = stronger regularization.C=0.01 → simple model. C=100 → complex, risk of overfit. Default 1.0 is a good start.
penalty='l2'Which regularization type to apply.l2 (default) shrinks all weights. l1 zeros out irrelevant ones. None = no regularization.
max_iter=1000Max gradient descent steps before stopping.If sklearn warns ConvergenceWarning, increase max_iter or scale your features.

The Decision Boundary

The prediction flips at σ(z) = 0.5, which means z = 0. So the decision boundary is wᵀx + b = 0 — always linear. Logistic regression cannot learn curved boundaries natively. For that you need polynomial features (L3) or a different model (L12+).

📊 Evaluation Metrics for Classification

Accuracy is a trap. Learn how to actually measure classifier performance.

Why Accuracy Alone Fails

Imagine a cancer detection model. Dataset: 950 healthy, 50 sick. A model that predicts "healthy" for everyone scores 95% accuracy — but it never catches a single sick patient. Accuracy hides what matters when classes are imbalanced. You need metrics that break down what kind of errors the model is making.

The Confusion Matrix

Predicted: PositivePredicted: Negative
Actual: PositiveTP — True Positive ✓FN — Missed it ✗
Actual: NegativeFP — False alarm ✗TN — True Negative ✓
False Positive (Type I) — False alarm

Predicted positive, actually negative. E.g. flagging a legitimate email as spam. Costly when action is expensive or annoying.

False Negative (Type II) — Missed case

Predicted negative, actually positive. E.g. missing a cancer diagnosis. Costly when missing the case has severe consequences.

For the "predict everyone healthy" model: TN=950, FP=0, FN=50, TP=0 — it never catches a single sick person despite 95% accuracy.

The Four Metrics

MetricFormulaThe question it answers
Accuracy(TP+TN) / totalOverall correctness — misleading when imbalanced
PrecisionTP / (TP+FP)Of everything I predicted positive, how many actually were? (trust my positives)
RecallTP / (TP+FN)Of all actual positives, how many did I catch? (miss rate)
F1 Score2·(P·R)/(P+R)Harmonic mean of both — penalises imbalance between them

How to Read F1 Score

F1 ranges from 0 to 1. But context matters more than the number — always compare against a sensible baseline.

F1Interpretation
≥ 0.90Excellent — production-ready for most tasks
≥ 0.80Good — acceptable for most use cases
≥ 0.70Mediocre — room for improvement
≥ 0.50Poor — barely better than naive guessing
< 0.50Bad — something is wrong
F1 punishes imbalance

Precision=0.99, Recall=0.10 → F1 ≈ 0.18 (heavily penalised). Precision=0.90, Recall=0.90 → F1 = 0.90. A high F1 means the model is neither spamming false alarms nor missing too many positives — it's doing both jobs simultaneously.

Precision–Recall Tradeoff

They pull in opposite directions as you move the threshold. There's no free lunch — choose based on the cost of each error type in your problem.

Lower threshold → higher Recall

Predict positive more often. Catch more true positives, but also more false alarms. Use when missing a positive is costly (cancer, fraud).

Higher threshold → higher Precision

Only predict positive when very confident. Fewer false alarms, but miss more real positives. Use when false alarms are costly (spam filters).

ROC Curve and AUC

The ROC curve sweeps the threshold from 0 → 1 and at each point plots False Positive Rate (x) vs True Positive Rate/Recall (y). AUC summarises the whole curve in one number.

AUCMeaning
1.0Perfect — ranks every positive above every negative
0.9+Excellent
0.8Good — ranks positives above negatives 80% of the time
0.5No better than random guessing
< 0.5Worse than random — flip your predictions

🌳 Decision Trees

Learning a flowchart from data — highly interpretable, but overfitting is the core enemy.

What is a Decision Tree?

A decision tree learns a series of if/else rules directly from data, forming a flowchart. Each internal node asks one question about one feature, each branch is the yes/no answer, and each leaf holds a class prediction.

Is duration < 180 seconds?
├── Yes → Is pdays = 999?
│         ├── Yes → Predict: No subscription (class 0)
│         └── No  → Predict: Subscription (class 1)
└── No  → Predict: Subscription (class 1)
No feature scaling needed

Decision trees never compare features against each other. Each split asks "is feature X above threshold T?" — one feature at a time, in its own units. Unlike KNN or logistic regression, scale differences between features are completely irrelevant.

Growing the Tree — Recursive Binary Splitting

The Algorithm

1. At the current node, try every possible split on every feature.
2. Choose the split with the lowest Gini impurity (or highest information gain).
3. Recurse on each child node.
4. Stop when a pruning criterion is hit (max depth, too few samples, etc.).

Gini Impurity — The Splitting Criterion

At every node the tree asks: how mixed are the classes here? Gini impurity measures that mixture. The tree always picks the split that minimises Gini.

G(S) = 1 − Σ pᵢ²

pᵢ = proportion of class i in the set S

SituationGiniMeaning
Pure node — all one class (p₁=1, p₂=0)0Perfect — no mixing
50/50 split (p₁=0.5, p₂=0.5)0.5Worst possible — maximum uncertainty
70/30 split0.42Partially separated

Lower Gini = purer node = better split. The tree picks whichever feature+threshold combination produces the lowest weighted Gini in the children.

Gini Gain — How the Split is Chosen

After splitting set S into children S₁ and S₂, the Gini gain is:

Gini Gain = G(S) − [ (|S₁|/|S|)·G(S₁) + (|S₂|/|S|)·G(S₂) ]

Weighted average impurity of children subtracted from parent impurity. Maximise this.

The tree tries every feature and every threshold, picks the combination with the highest Gini gain — meaning the children are as pure as possible.

Information Gain — The Alternative

Instead of Gini, you can measure node impurity using Entropy. The goal flips: maximise information gain (reduction in entropy).

Entropy formula

H(S) = −Σ pᵢ log₂(pᵢ). Zero for a pure node, 1.0 for a 50/50 split. Use criterion='entropy' in sklearn.

Gini vs Entropy in practice

Both produce very similar trees. Gini is cheaper (no log). Entropy gives slightly better probability calibration. Gini is the sklearn default and the standard choice.

Summary rule: Gini → minimise (lower = purer). Information Gain → maximise (higher = better split). Both find the same splits — they are just two ways to measure the same thing.

Overfitting — The Core Problem

A fully grown tree (no constraints) creates a near-unique path for every training point. It memorises the data — including noise and outliers — rather than learning general patterns.

Symptoms of Overfitting

Training accuracy ≈ 100%

The tree has a leaf for every training point (or near it). Getting 100% on training data is trivially easy — it tells you nothing about generalisation.

Large train/test gap

Test accuracy is significantly lower than training accuracy (e.g. 99% train vs 71% test). The model learned the quirks of the training set, not the real pattern.

Poor minority class recall

On imbalanced data, the tree learns to predict the majority class almost always. High accuracy, but terrible recall on the class that matters.

Extremely deep tree

Visualising the tree shows hundreds of levels. A well-generalising tree is usually readable — not a wall of branches.

How to Fix Overfitting

ParameterWhat it controlsEffect
max_depthMaximum levels from root to leafMost important lever. Start with 3–6, tune with CV.
min_samples_splitMin samples needed to split a nodeHigher value → fewer splits → simpler tree
min_samples_leafMin samples required at each leafPrevents tiny leaves that memorise one or two points

Finding the Best Depth — Train vs Validation Plot

Sweep max_depth from 1 to 20. Train accuracy climbs monotonically toward 100%. Validation accuracy climbs, peaks, then falls — the peak is your optimal depth.

depths = np.arange(1, 21)
train_accs, val_accs = [], []

for d in depths:
    t = DecisionTreeClassifier(max_depth=d, random_state=42)
    t.fit(X_train, y_train)
    train_accs.append(t.score(X_train, y_train))
    val_accs.append(t.score(X_test, y_test))

plt.plot(depths, train_accs, label='Train')
plt.plot(depths, val_accs,   label='Validation')
plt.xlabel('max_depth'); plt.legend(); plt.show()

best_depth = depths[np.argmax(val_accs)]
print(f"Best depth: {best_depth}")

Reading the plot: Both lines low → underfitting (increase depth). Big gap between train and validation → overfitting (decrease depth). Lines close together and decent → good generalisation.

dtc.set_params(max_depth=6, min_samples_split=20, class_weight="balanced")
dtc.fit(X_train, y_train)

print(classification_report(y_test, dtc.predict(X_test)))

What changes: max_depth=6 stops the tree from memorising noise. min_samples_split=20 prevents splits on tiny groups. class_weight='balanced' automatically increases the penalty for misclassifying the minority class — the tree can no longer ignore class 1 and get away with it. Class 1 recall improves significantly.

    'max_depth':        [3, 4, 7],

What's happening: GridSearchCV tries all 3×3 = 9 combinations of max_depth and min_samples_split. For each combination it runs 10-fold CV on the training data — fitting and evaluating 90 trees in total. It returns the combination with the best average validation score. **gridCV.best_params_ unpacks the dictionary directly into set_params().

🌲 Random Forests

Turn one unstable tree into hundreds of stable ones — ensembling at its best.

The Core Idea: Bagging

Bootstrap Aggregating (Bagging): train many models on different random samples of the data, then combine their predictions. Diversity between models is the key.

Why does bagging work?

Each individual tree overfits differently. When you average many differently-overfitted models, the errors cancel out and the signal remains. Variance drops dramatically.

Random Forests = Bagging + Feature Randomness

Random Forests add one extra step: at each split, only a random subset of features is considered. This decorrelates the trees — if one feature dominates, not all trees will rely on it.

Classification vote

Each tree predicts a class. The majority vote across all trees is the final prediction.

Regression average

Each tree predicts a number. The mean across all trees is the final prediction.

📍 K-Nearest Neighbours (KNN)

No training, no model — just memory and distance.

💡 KNN is a discriminative classifier — it directly models P(y|x) by looking at the class labels of nearby points.

How KNN Works

KNN is the simplest classifier conceptually: to classify a new point, find its K nearest training points and take a majority vote.

Step by step

1. Compute the distance from the new point to every training point.
2. Sort all training points by distance — closest first.
3. Select the K closest neighbours.
4. Predict the majority class among those K neighbours.

No model is built. No parameters are learned. The training data is the model — fit() just stores it.

KNN Gives You Probabilities, Not Just a Label

The vote counts translate directly into class probabilities. For k=5 and neighbours [A, A, C, B, C]:

ClassVotesProbability
A22/5 = 0.40
B11/5 = 0.20
C22/5 = 0.40

A 40/40/20 split tells you the model is uncertain between A and C. A 5/0/0 split tells you it is completely confident. Use predict_proba() to access these.

Distance Metrics

KNN needs a way to measure "closeness" between points. The three main options:

MetricFormulaIntuition
Euclidean√Σ(xᵢ − yᵢ)²Straight-line distance — most common, default
ManhattanΣ|xᵢ − yᵢ|Grid-travel distance, like city blocks — better when data has many dimensions or outliers
Minkowski(Σ|xᵢ−yᵢ|ᵖ)^(1/p)Generalisation — Euclidean when p=2, Manhattan when p=1

⚠️ Critical: Feature Scaling is Mandatory

KNN decides "nearest" purely by distance. Without scaling, large-range features completely drown out small-range ones.

Example: Age (0–80) vs Income (0–500,000)

Distance from Person A (age=25, income=30,000) to Person B (age=26, income=80,000):

√((26−25)² + (80000−30000)²) = √(1 + 2,500,000,000) ≈ 50,000

The age difference of 1 year contributes almost nothing. Income dominates entirely. KNN behaves as if age doesn't exist.

After scaling to [0,1]: age=0.31 vs 0.33, income=0.06 vs 0.16 — both features contribute fairly.

⚠️ Always normalise features to [0,1] with MinMaxScaler or standardise with StandardScaler before using KNN. No exceptions.

Choosing K — The Bias-Variance Tradeoff

K is a hyperparameter — you set it manually before training. It controls the fundamental character of the model.

K valueBiasVarianceBoundaryRisk
K = 1Very lowVery highJagged, wraps every pointOverfitting
K = 5–20Low–ModerateModerateSmooth — usually best rangeSweet spot
K = N (all)Very highVery lowAlways predicts majority classUnderfitting

Tune K with cross-validation. Use odd K for binary problems to avoid ties.

The K=1 Training Accuracy Trap

Warning: 100% training accuracy is meaningless with K=1

When K=1, every training point classifies itself — its nearest neighbour is itself (distance = 0). This guarantees 100% accuracy on training data regardless of how good or bad the model is.

This is the definition of overfitting. Always evaluate on a held-out test set the model has never seen.

Why KNN is a "Lazy Learner"

There is no training step. KNN simply stores all training data. All computation — finding neighbours, voting — happens at prediction time. This means:

Training

Instant — just store the data. fit() does nothing but memorise.

Prediction

Slow for large datasets — must compute distance to every training point for each new input.

The Curse of Dimensionality

KNN degrades badly as the number of features grows. Here's why:

In high-dimensional space, all points become approximately the same distance from each other. The maximum and minimum distances converge:

DimensionsNearest neighbour distanceFarthest neighbour distanceResult
2D0.19.9"Nearest" is meaningful
100D9.810.0"Nearest" is meaningless

KNN's entire logic — nearby points share the same class — breaks down when "nearby" stops being informative.

Fix: Use feature selection or PCA to reduce dimensions before applying KNN.

KNN: Strengths and Limitations

Strengths

  • Simple and intuitive — no training needed
  • Produces probability estimates naturally
  • Works well when classes are well-separated
  • Non-linear decision boundary automatically

Limitations

  • Slow at prediction time on large datasets
  • Fails in high dimensions (curse of dimensionality)
  • Sensitive to irrelevant/unscaled features
  • Poor with discrete/categorical features

↔️ Support Vector Machines (SVM)

Find the boundary that keeps classes as far apart as possible — and prove it's the best possible one.

Why SVM Instead of Logistic Regression?

Logistic regression also draws a decision boundary — so why use SVM at all? The key difference: logistic regression finds a boundary, SVM finds the best boundary.

●  ●              ○  ○
  ●      ???      ○       ← many valid lines exist in this gap
●  ●              ○  ○

Logistic Regression → wherever gradient descent stops (somewhat arbitrary)
SVM               → the unique line sitting in the exact middle of the widest gap
Better generalisation on small datasets

Logistic regression's boundary could sit close to one class. SVM's boundary is provably as far as possible from both — a test point near the boundary is more likely to be classified correctly.

Handles non-linear patterns

Logistic regression is stuck in the original feature space unless you manually engineer features. SVM uses the kernel trick to implicitly work in a higher-dimensional space where non-linear data becomes linearly separable.

Works in high-dimensional spaces

In text classification (10,000+ features, maybe 1,000 documents), logistic regression tends to overfit. SVM's sparse solution — only support vectors matter — naturally resists this.

Globally optimal solution

Margin maximisation is a convex problem — one unique answer, always. Logistic regression can get stuck or vary based on initialisation and learning rate.

SituationSVM advantage over Logistic Regression
Small datasetMax-margin boundary generalises better
High-dimensional features (text, genomics)Sparse solution, less overfitting
Non-linear patterns in dataKernel trick — no manual feature engineering needed
Noisy overlapping classesSoft margin is more principled than log-loss
Large, clean, linearly separable dataBoth give similar results — logistic regression is faster
On large clean datasets with linear separability, logistic regression and linear SVM perform almost identically. The SVM advantage shows most clearly when data is limited, high-dimensional, or non-linearly separable.

The Problem With Other Boundaries

For a linearly separable dataset, there are infinitely many valid hyperplanes that correctly separate two classes. Logistic Regression picks one via gradient descent — it works, but the choice is somewhat arbitrary. SVM asks a sharper question:

SVM's core question

Of all the valid decision boundaries, which one is the most confident? The answer: the one furthest from all training points — the maximum-margin hyperplane.

The Margin — What It Is and Why It Matters

The margin is the perpendicular distance between the decision boundary and the nearest training points from each class. Visualise it as a "no-man's-land" strip around the boundary:

Class A (●)    Margin edge     Boundary     Margin edge    Class B (○)
    ●              |               |               |            ○
      ●       ←───|───────────────|───────────────|───→        ○
    ●              |               |               |            ○
                       ← margin ←   → margin →
                          (maximise the total width)

A wider margin means the model has a large safe zone around its decision boundary. Test points that land in this zone are classified confidently. A narrow margin means the boundary is dangerously close to training points — small noise in new data tips it the wrong way.

The maths behind the margin

The hyperplane is w · x + b = 0. The two margin edges sit at w · x + b = +1 and w · x + b = −1. The margin width = 2 / ‖w‖. Maximising the margin means minimising ‖w‖ — a convex quadratic optimisation with a unique global solution.

Support Vectors — The Only Points That Matter

The support vectors are the training points that sit exactly on the margin edges (where w · x + b = ±1). These are the closest points to the boundary from each class.

Key insight: only the support vectors define the decision boundary. You could delete every other training point and the model would be identical. The entire SVM solution depends on just these few critical points.

This is unlike logistic regression or neural networks where every training point influences the final model. SVM is a sparse solution — usually only 5–30% of training points end up as support vectors.

Why Use SVM? Advantages Over Other Models

Works in high-dimensional spaces

SVM is effective even when there are more features than training examples — common in text classification and genomics. Most other models struggle here.

Memory efficient

Only support vectors are stored after training — a small fraction of the dataset. Logistic regression stores full weight vectors; SVMs are compact.

Globally optimal solution

The margin maximisation is a convex optimisation — no local minima. You always get the same best answer, unlike neural networks where random initialisation matters.

Robust to outliers

Non-support-vector points have zero influence. Outliers far from the boundary don't distort the model (unlike linear regression where every point pulls the fit).

When SVM Is the Best Choice

ScenarioWhy SVM wins
Small-to-medium dataset (<50k rows)Finds the globally optimal boundary; other models need more data to converge reliably
High-dimensional features (text, gene expression)Linear SVM handles more features than samples; sparse solution avoids overfitting

You need a theoretically grounded boundaryMaximum margin has statistical learning theory backing (VC dimension bounds)

Soft Margin SVM and the C Parameter

Real data overlaps — a "hard margin" that allows zero misclassifications either fails or overfits to noise. Soft margin SVM allows some points to violate the margin, but penalises each violation. The parameter C controls this trade-off:

C valueMarginViolations allowedRisk
Small (0.01)WideManyUnderfitting — boundary too simple
Default (1.0)BalancedSomeGood starting point
Large (1000)NarrowVery fewOverfitting — boundary wraps training data

Think of C as the inverse of regularisation — exactly like the C parameter in logistic regression. Higher C = less regularisation = more complex boundary = risk of overfitting.

Overfitting symptom with high C

99% training accuracy, 70% test accuracy. The boundary has contorted to classify nearly every training point correctly — including noise. Fix: lower C and use GridSearchCV to find the right value.

The Kernel Trick — Non-Linear Boundaries

Most real data isn't linearly separable. The solution: map data to a higher-dimensional space where it becomes linearly separable, then fit a standard SVM there.

# 1D — not separable:   ● ○○ ●
# Map to 2D (x → x²):  ●        ●
#                          ○○
# Now a straight line separates them

The kernel trick: you never compute the transformation φ(x) directly. Instead, the kernel function K(x, x') = φ(x)·φ(x') computes the dot product in the high-dimensional space using only the original features. This is computationally cheap but gives the power of a high-dimensional separation.

Common Kernels

γ controls how far each support vector's influence reaches in the RBF kernel:

High γ makes each support vector act like a spike — the boundary wraps tightly around training points. Low γ makes each support vector cast a wide net — the boundary is smooth and general. Always tune C and γ together. Use gamma='scale' as a safe default (sklearn sets γ = 1 / (n_features × variance)).

The Scaling Requirement

SVM computes distances between points (‖x − x'‖). If one feature has values in [0, 1000] and another in [0, 1], the first dominates every distance calculation and the second becomes invisible. Always scale before SVM — wrap it in a Pipeline so scaling is never accidentally skipped.

Full sklearn Code

from sklearn.svm import SVC
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import Pipeline
from sklearn.model_selection import train_test_split, GridSearchCV
from sklearn.metrics import classification_report

# Step 1: Always scale inside a Pipeline — prevents data leakage
pipeline = Pipeline([
    ('scaler', StandardScaler()),
    ('svm',    SVC(kernel='rbf', probability=True))
])

# Step 2: Tune C and gamma together — they interact
param_grid = {
    'svm__C':     [0.1, 1, 10, 100],
    'svm__gamma': ['scale', 0.001, 0.01, 0.1]
}
grid = GridSearchCV(pipeline, param_grid, cv=5, scoring='f1', n_jobs=-1)
grid.fit(X_train, y_train)

# Step 3: Evaluate on held-out test set
best = grid.best_estimator_
print(classification_report(y_test, best.predict(X_test)))
print("Best params:", grid.best_params_)
# For large datasets — use LinearSVC instead
from sklearn.svm import LinearSVC

pipeline_linear = Pipeline([
    ('scaler', StandardScaler()),
    ('svm',    LinearSVC(C=1.0, max_iter=2000))
])
pipeline_linear.fit(X_train, y_train)
print(pipeline_linear.score(X_test, y_test))

Key Rules Card

SVM in practice — 5 rules
  1. Always scale — margin is distance-based; unscaled features destroy it
  2. Use a Pipeline — scaler must be fit on training data only
  3. Tune C and γ together — they interact; grid search both simultaneously
  4. Start with RBF — if that fails try linear; only try polynomial for specific reasons
  5. Switch to LinearSVC at scale — SVC is O(n²–n³), brutal on large datasets

🔄 Model Validation

Cross-validation, GridSearchCV, and Pipelines — how to get an honest estimate of model performance.

Why Validation Matters

You build a model and measure its accuracy. That number only means something if it was measured on data the model never saw during training. The more your model (or your tuning) interacts with the test set, the more optimistic your results become — and the less trustworthy they are on truly unseen data.

Bias and Variance — The Fundamental Tradeoff

Every model's prediction error has three components:

Total Error = Bias² + Variance + Irreducible Noise

Irreducible noise is random noise in the data — you cannot fix it. Focus on Bias and Variance.

Bias (Underfitting)Variance (Overfitting)
What it isModel too simple — misses the real patternModel too complex — memorises noise
SymptomPoor training AND test accuracyHigh training accuracy, much lower test accuracy
FixMore complex model, more featuresRegularisation, pruning, more data, CV tuning
ExampleDecision tree max_depth=1Decision tree max_depth=None

Three-Way Data Split

For serious work, split your data three ways — not two. The test set is a vault you open exactly once at the very end.

# Full Dataset
# ├── Training set   (70–90%) → model learns from this
# ├── Validation set  (5–15%) → tune hyperparameters here (open repeatedly)
# └── Test set        (5–15%) → final honest evaluation — touch ONCE

from sklearn.model_selection import train_test_split

# Step 1: carve off 20% as the held-out test set
x_train, x_test, y_train, y_test = train_test_split(x, y, train_size=0.8)

# Step 2: split that 20% in half → 10% validation, 10% test
x_val, x_test, y_val, y_test = train_test_split(x_test, y_test, test_size=0.5)

print(len(x_train), len(x_val), len(x_test))  # e.g. 3681, 460, 460

The discipline: Every time you tune a hyperparameter based on test performance, you corrupt that number. The validation set is your proxy test set — use it for all tuning. The real test set is seen only at the end to report your final result.

Checking Class Balance Across Splits

Always verify that your class ratio is preserved across splits — especially important for imbalanced data like spam detection.

# value_counts(normalize=True) shows proportions instead of raw counts
print(f"% spam in all data:        {data['class'].sum()/len(data):.3f}")
print(f"% spam in training data:   {y_train.sum()/len(y_train):.3f}")
print(f"% spam in validation data: {y_val.sum()/len(y_val):.3f}")
print(f"% spam in test data:       {y_test.sum()/len(y_test):.3f}")

# Or use value_counts for non-binary targets
print(y_train.value_counts(normalize=True))

What you're checking: If 39% of the full dataset is spam, each split should also be roughly 39% spam. A random split can accidentally put most rare examples in one set. Use stratify=y in train_test_split to guarantee this.

Data Leakage in Preprocessing

The most common mistake: fitting a scaler on the full dataset before splitting. The scaler learns the mean and std of test rows — those rows then influenced the transformation applied to them. The test set is no longer truly unseen.

# ❌ WRONG — scaler has seen test data (data leakage)
scaler = StandardScaler()
X_scaled = scaler.fit_transform(X)          # test rows included in mean/std
X_train, X_test = train_test_split(X_scaled)

# ✅ CORRECT — scaler only sees training data
X_train, X_test = train_test_split(X)
scaler = StandardScaler()
X_train = scaler.fit_transform(X_train)     # fit on train only
X_test  = scaler.transform(X_test)          # apply same transform, no refit
⚠️ The same rule applies to any preprocessing that learns from data: imputation (mean/median), PCA, feature selection, one-hot encoding statistics. All must fit on training data only.

K-Fold Cross-Validation

A single train/test split can be lucky or unlucky. K-fold CV gives a reliable estimate by rotating which part of the data is the test set — every point gets tested exactly once.

# K=5 illustration:
# Fold 1: [TEST ][train][train][train][train] → score 1
# Fold 2: [train][TEST ][train][train][train] → score 2
# Fold 3: [train][train][TEST ][train][train] → score 3
# Fold 4: [train][train][train][TEST ][train] → score 4
# Fold 5: [train][train][train][train][TEST ] → score 5
# Final = mean(scores) ± std(scores)
Standard KFold

Splits data into k equal folds randomly. Fast. Use for regression or balanced classification.

StratifiedKFold

Each fold preserves the same class proportion as the full dataset. Essential for imbalanced classification — ensures rare classes appear in every fold.

Typical choices: k=5 or k=10. Larger k = less biased estimate, more compute. Standard deviation across folds tells you how stable the model is — high std means results depend heavily on which data it saw.

cross_validate — All Metrics in One Call

Instead of writing loops manually, cross_validate handles splitting, training, and evaluating for you across all folds.

from sklearn.model_selection import cross_validate, StratifiedKFold
from sklearn.neighbors import KNeighborsClassifier
import pandas as pd

model = KNeighborsClassifier()
my_cv  = StratifiedKFold(n_splits=5)

# Define all the metrics you want measured per fold
my_metrics = {
    'accuracy':  'accuracy',
    'precision': 'precision_macro',
    'recall':    'recall_macro',
    'f1':        'f1_macro'
}

scores = cross_validate(model, x, y, cv=my_cv, scoring=my_metrics)

# Put results in a DataFrame — one row per fold
df = pd.DataFrame(scores)
print(df[['test_accuracy', 'test_precision', 'test_recall', 'test_f1']])

What's happening: You never call .fit() or .predict() yourself. cross_validate does all of it — splits data, trains on k-1 folds, evaluates on the remaining fold, repeats k times. Returns a dictionary of arrays (one value per fold per metric). _macro means average the metric equally across all classes regardless of class size.

Reporting CV Results — Mean ± Std

With k folds you have k scores, not one. Always report both the mean and standard deviation.

# Simple mean/std summary
scores_df = pd.DataFrame(scores).drop(['fit_time', 'score_time'], axis=1)

Reading the results: A model with mean=0.88, std=0.02 is more trustworthy than one with mean=0.91, std=0.12. The first is consistently good. The second is erratic — its performance depends heavily on which data it happened to train on. High std is a warning sign worth investigating before trusting the mean.

GridSearchCV — Hyperparameter Tuning with CV

Never tune hyperparameters against your test set. GridSearchCV does all tuning inside the training data using cross-validation, then you evaluate the winner on the test set once at the end.

from sklearn.model_selection import GridSearchCV, StratifiedKFold
from sklearn.neighbors import KNeighborsClassifier

model  = KNeighborsClassifier()
my_cv  = StratifiedKFold(n_splits=5)

# 5 values × 1 hyperparameter × 5 folds = 25 models trained
hyperparams = {'n_neighbors': [1, 2, 3, 4, 5]}

# refit=False when using multiple scoring metrics
gridsearch = GridSearchCV(
    estimator=model,
    param_grid=hyperparams,
    scoring={'accuracy': 'accuracy', 'f1': 'f1_macro'},
    cv=my_cv,
    refit=False
)
gridsearch.fit(x, y)

# Results: mean score per metric per hyperparameter combination
grid_df = pd.DataFrame(gridsearch.cv_results_)
print(grid_df[['param_n_neighbors', 'mean_test_accuracy', 'mean_test_f1']])

What's happening: GridSearchCV trains a model for every combination in your grid, evaluates each with k-fold CV, and stores all results in cv_results_. refit=False is needed when you pass multiple scoring metrics — sklearn can't automatically pick a "best" model when there are multiple scores. When you use a single metric, omit refit=False and you get best_estimator_ directly.

Getting the Best Model (Single Metric)

# With a single scoring metric, GridSearchCV refits the best model automatically
gridsearch = GridSearchCV(
    KNeighborsClassifier(),
    {'n_neighbors': [2, 3, 4, 5]},
    cv=StratifiedKFold(5),
    scoring='f1_macro'     # single metric → best_estimator_ available
)
gridsearch.fit(x, y)

print(gridsearch.best_estimator_)   # best model, ready to use
print(gridsearch.best_params_)      # {'n_neighbors': 4}
print(gridsearch.best_score_)       # best mean CV F1

# Final evaluation on the held-out test set
print(classification_report(y_test, gridsearch.predict(x_test)))

How many models does GridSearchCV train?

Combinations × folds. Tuning 3 hyperparameters with 5 values each, 10-fold CV: 5³ × 10 = 1,250 models. With 6 hyperparameters at 10 values: 10⁶ × 10 = 10 million. That's when RandomizedSearchCV becomes necessary.

sklearn Pipeline — Preventing Leakage Inside CV

When GridSearchCV runs, it fits your model on each training fold. But if your scaler was fitted before GridSearchCV, it already saw all folds — including the test fold for each iteration. Contaminated.

The fix: wrap preprocessing and model in a Pipeline. GridSearchCV refits the entire pipeline on each training fold, guaranteeing the scaler never sees test-fold data.

from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression

pipe = Pipeline([
    ('scaler', StandardScaler()),    # step 1 — name: 'scaler'
    ('model',  LogisticRegression())  # step 2 — name: 'model'
])

# Tune model parameters using __ (double underscore) to reference inside pipeline
grid = GridSearchCV(pipe, {'model__C': [0.01, 0.1, 1, 10]}, cv=5, scoring='f1')
grid.fit(X_train, y_train)

# Final evaluation
print(classification_report(y_test, grid.predict(X_test)))

What's happening: Inside each CV fold, pipe.fit(X_train_fold) triggers scaler.fit_transform(X_train_fold) then model.fit(X_train_scaled). The held-out fold only ever sees scaler.transform() — never fit_transform(). The model__C syntax means "the parameter C of the step named model".

from sklearn.model_selection import train_test_split, GridSearchCV, StratifiedKFold
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression
from sklearn.metrics import classification_report
import pandas as pd

X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)

# Step 2 — build Pipeline (scaler + model together)
pipe = Pipeline([
    ('scaler', StandardScaler()),
    ('model',  LogisticRegression(max_iter=1000))
])

# Step 3 — tune hyperparameters inside training data only
param_grid = {'model__C': [0.01, 0.1, 1, 10, 100]}
grid = GridSearchCV(pipe, param_grid, cv=StratifiedKFold(5), scoring='f1')
grid.fit(X_train, y_train)

print("Best params:", grid.best_params_)
print("Best CV F1:",  grid.best_score_)

# Step 4 — final evaluation on the held-out test set (do this ONCE)
print(classification_report(y_test, grid.predict(X_test)))

The four rules of this workflow:
1. Split train/test first, before anything else.
2. All preprocessing is inside a Pipeline — never fitted on the full dataset.
3. All tuning happens inside GridSearchCV using only training data.
4. The test set is evaluated exactly once at the very end. Its score is your final reported result.

🎯 Multiclass Classification

Predicting one of three or more classes — and how to extend binary classifiers to do it.

Binary vs Multiclass

Everything learned so far (logistic regression, SVM, naive bayes) was designed for binary classification — exactly 2 classes. Multiclass means predicting one of 3 or more classes:

Binary examples

Spam / not spam · Delayed / not delayed · Cancer / no cancer

Multiclass examples

Glass type (6 classes) · Handwritten digit (0–9) · Disease diagnosis (healthy / condition A / B / C)

The challenge: most classifiers output a single score (class 0 vs class 1). Extending to K classes requires a strategy.

Two Types of Models

Native multiclass (just works)Why
Decision TreeEach leaf can be any class — splits recursively on all K classes
KNNMajority vote among K neighbours — vote goes to any class
Random ForestEnsemble of trees — same reason as Decision Tree
Naive BayesComputes P(class|features) for each class, picks highest
Binary-only (needs a strategy)Default sklearn strategy
Logistic RegressionOne-vs-Rest (OvR)
SVM (LinearSVC)One-vs-Rest (OvR)

Strategy 1: One-vs-Rest (OvR)

Train K separate binary classifiers, one per class. Each answers: "Is this example my class, or NOT my class?" At prediction time, all K classifiers give a confidence score — whichever is most confident wins.

# 6 glass types → 6 classifiers:
Classifier 1 (containers):  "I'm 15% confident"
Classifier 2 (tableware):   "I'm  8% confident"
Classifier 3 (headlamps):   "I'm 72% confident"  ← highest → prediction
Classifier 4 (vehicle):     "I'm 31% confident"
Classifier 5 (float):       "I'm 44% confident"
Classifier 6 (non-float):   "I'm 20% confident"

Final prediction → headlamps
These scores don't sum to 1 — you're doing a relative comparison (who is most confident), not computing true probabilities. That's why softmax (multinomial LR) is sometimes better for logistic regression.

Strategy 2: One-vs-One (OvO)

Train one binary classifier for every pair of classes. For K classes: K(K−1)/2 classifiers. Each test example goes to all classifiers — the class that wins the most head-to-head duels wins overall.

# 6 classes → 6×5/2 = 15 classifiers, e.g.:
containers vs tableware   → containers wins
containers vs headlamps   → headlamps wins
headlamps  vs vehicle     → headlamps wins
headlamps  vs float       → headlamps wins
... (all 15 duels)

Win counts: headlamps=4, float=4, containers=3, tableware=3 ...
Final prediction → headlamps (or float — tie broken by confidence margin)

Why OvO can be better: each classifier only ever sees 2 balanced classes — no "1 class vs all others" imbalance. Classes are learned more cleanly.

Evaluating Multiclass Models

Accuracy collapses everything to one number — it hides which classes are failing. You need per-class performance.

Confusion Matrix (K × K)

Rows = true class, columns = predicted class. Diagonal = correct predictions. Off-diagonal cells reveal which classes get confused with each other.

#          Predicted →
#       cont tab head veh flt non
cont  [ 12    0    0    1    0    0 ]  ← 1 container predicted as vehicle
head  [  0    0    8    0    0    2 ]  ← 2 headlamps confused with non-float

A perfect model has non-zero values only on the diagonal.

classification_report for Multiclass

#               precision  recall  f1-score  support
containers         0.80     0.92      0.86       13

Macro avg treats all classes equally — harsh when classes are imbalanced (rare class drags down the score). Weighted avg weights by class size — more realistic overall picture.

Multiclass in practice — 5 rules

Model Deep Dives

Naive Bayes (L9b)

What it does: Uses Bayes' Rule to compute P(class|features) ∝ P(features|class) × P(class). Assumes all features are conditionally independent given the class (the "naive" assumption).

Gaussian NB: assumes each feature follows a normal distribution within each class. Estimates mean and variance per feature per class from training data.

Key params: var_smoothing — adds a small value to variances to avoid zero-probability issues (like Laplace smoothing for continuous features).

Pros: Fast to train and predict · Works well on small datasets · Handles high-dimensional text features · Native multiclass

Cons: Naive independence assumption is rarely true in practice · Can be overconfident · Poor probability calibration

Logistic Regression (L10)

What it does: Passes a linear combination of features through the sigmoid function σ(z) = 1/(1+e⁻ᶻ) to output a probability between 0 and 1. Optimises cross-entropy loss via gradient descent.

Key params: C (inverse regularisation — higher C = less regularisation = risk of overfitting) · penalty ('l2' shrinks all weights, 'l1' sets small weights to zero — feature selection) · max_iter (increase if convergence warning)

Pros: Fast · Probabilistic output · Interpretable coefficients · Good baseline · Works well on linearly separable data

Cons: Can only learn linear decision boundaries · Needs scaling · Struggles with highly non-linear patterns

Decision Trees (L12)

What it does: Recursively splits data using the feature+threshold that best reduces impurity. Each internal node is a question, each leaf is a prediction.

Splitting criteria: Gini impurity (minimise) — default in sklearn. Information gain/entropy (maximise) — alternative. Both give similar results.

Gini impurity: 1 − Σpᵢ² — 0 means a pure node (all one class), 0.5 is maximally impure (50/50 split for binary).

Key params: max_depth · min_samples_split · min_samples_leaf · class_weight='balanced'

Overfitting symptoms: 100% training accuracy + low test accuracy · Very deep tree · Poor recall on minority class

Pros: No scaling needed · Interpretable · Handles mixed types · Native multiclass

Cons: High variance — small data changes cause very different trees · Overfits without constraints

Random Forest (L13)

What it does: Trains many decision trees, each on a random bootstrap sample of data and a random subset of features at each split. Prediction = majority vote across all trees.

Why it works: Each tree is high-variance but low-bias. Averaging many uncorrelated trees cancels out individual errors — the ensemble is stable.

Key params: n_estimators (more trees = better, up to diminishing returns, never overfits) · max_depth · max_features (feature randomness) · class_weight

Pros: Low variance · Robust to overfitting · feature_importances_ built-in · No scaling needed · Native multiclass

Cons: Slow to predict · Less interpretable than a single tree · Large memory footprint

KNN (L14)

What it does: Stores all training data. For a new point, finds the K nearest training examples (by Euclidean distance) and takes a majority vote. No explicit training step — "lazy learner".

Key params: n_neighbors K (small K = overfit / jagged boundary, large K = underfit / smooth boundary) · metric (euclidean, manhattan)

Curse of dimensionality: In high dimensions, all points become approximately equidistant — "nearest" stops being meaningful. KNN degrades above ~20 features.

K=1 training accuracy: Always 100% (each point is its own nearest neighbour) — meaningless, never use training accuracy with KNN.

Pros: No training · Simple · Non-linear boundary · Native multiclass · Probability output via vote proportions

Cons: Slow prediction O(n) · Needs scaling · Fails in high dimensions · Large memory (stores all training data)

SVM (L15)

What it does: Finds the hyperplane that maximises the margin — the perpendicular gap between the boundary and the nearest training points (support vectors). Only support vectors define the model.

C parameter: Inverse of regularisation. High C = narrow margin, few violations, risk of overfitting. Low C = wide margin, more violations, simpler model.

Kernel trick: Maps data to a higher-dimensional space where it's linearly separable — without computing the transformation directly. RBF kernel: K(x,x') = exp(−γ‖x−x'‖²).

Gamma (RBF only): High γ = narrow influence per point → wiggly boundary → overfit. Low γ = wide influence → smooth boundary → underfit. Always tune C and γ together.

Pros: Globally optimal solution · Works in high dimensions · Memory efficient (only support vectors) · Kernel trick handles non-linearity

Cons: O(n²–n³) — very slow on large data · Needs scaling · No probability output without probability=True (slower) · Hard to interpret

Model Validation (L16)

Bias–Variance tradeoff: Total Error = Bias² + Variance + Irreducible Noise. High bias = underfitting (model too simple). High variance = overfitting (model memorises training data).

K-fold cross-validation: Split data into K folds. Train on K−1, validate on 1. Rotate K times. Report mean ± std of the metric. Use StratifiedKFold for classification (preserves class balance in each fold).

GridSearchCV: Exhaustively tries all hyperparameter combinations × K folds. n_jobs=-1 runs in parallel. Use refit=False when scoring with multiple metrics.

Pipeline rule: Always wrap scaler + model in a Pipeline before GridSearchCV. Prevents data leakage — scaler is refit on each training fold only.

Data leakage: Using information from the test set during training (e.g., fitting a scaler on all data before splitting). Produces optimistic metrics that don't hold on new data.

Multiclass Classification (L17)

Native multiclass: Decision Tree, KNN, Random Forest, Naive Bayes — no strategy needed.

One-vs-Rest (OvR): K binary classifiers, one per class. At prediction, highest confidence score wins. Default in sklearn for LR and SVM.

One-vs-One (OvO): K(K−1)/2 classifiers, one per pair. Each is a head-to-head duel — class with most wins is predicted. Cleaner learning per classifier (balanced 2-class problem).

Confusion matrix: K×K grid. Rows = true class, columns = predicted. Diagonal = correct. Off-diagonal reveals which classes are confused with each other.

Macro avg: equal weight per class. Weighted avg: weighted by class size. Big gap between them → rare classes are being missed.

The Golden Rules

Rules that apply to every classification problem
  1. Scale before KNN, SVM, and Logistic Regression — all three are distance- or gradient-based
  2. Always use a Pipeline when combining preprocessing with CV or GridSearchCV — prevents data leakage
  3. Never evaluate on training data — use a held-out test set or cross-validation
  4. Check class balance first — if imbalanced, use class_weight='balanced' and report F1/recall, not just accuracy
  5. Look at per-class metrics — a high overall accuracy can hide a model that completely misses a rare class
  6. Tune C and gamma together for SVM — they interact; tuning one without the other is unreliable
  7. Decision Trees overfit by default — always constrain with max_depth or min_samples_split
  8. Use StratifiedKFold for classification — standard KFold can produce folds with missing classes
